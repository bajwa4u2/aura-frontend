### Phase 4 — APNs backend readiness
🟢 Audited and corrected. Key finding below; details in §5.

> **Architecture finding:** the iOS client registers **FCM** tokens
> (`provider=FCM`, `FirebaseMessaging.getToken()`), so live iOS push is
> delivered **FCM → APNs**, handled by `FcmPushAdapter` using `FIREBASE_*`
> credentials. The direct `ApnsPushAdapter` (`APPLE_APNS_*`) only fires for
> devices that register a *raw* APNs token — which the current client does
> not do. The `APPLE_APNS_*` Railway vars are therefore **dormant** today;
> they are wired correctly for completeness, but **iOS push working on
> TestFlight depends on the Firebase path**, not on `APPLE_APNS_*`.

### Phase 5 — iOS client push readiness
🟢 Audited; no code changes required. The Flutter client is already complete:
- Permission request — `FirebaseMessaging.requestPermission()` in
  `DeviceService` (boot path + explicit security-screen path).
- Token handling — `getToken()` → `POST /v1/devices/register`; `onTokenRefresh`
  re-syncs; auth-gated to avoid 401 spam.
- Lifecycle — registers on boot/login, `refreshPresence()` on resume (30-min
  debounce), `revokeCurrentDevice()` on logout, re-registers on next login.
- Tap routing — `NotificationBridge` handles foreground / background
  (`onMessageOpenedApp`) / killed-app (`getInitialMessage`) and routes calls,
  threads, and terminal-call payloads correctly.
- Call/ring — `_isCallInterrupt` forces a notification refresh so the incoming
  call overlay appears; silent `CALL_CANCELLED` dismisses the ring.

## 7. Remaining manual steps

These require macOS / the Apple Developer portal / the Codemagic dashboard /
a physical device and were **not** performed here.

### A. Apple Developer portal (Phase 3)
1. Confirm an **explicit App ID** `org.auraplatform.app` exists.
2. On that App ID enable capabilities: **Push Notifications** and
   **Associated Domains** (both are declared in `Runner.entitlements`; the
   provisioning profile must carry them or the archive codesign step fails).
3. Confirm an **App Store Connect app record** exists for
   `org.auraplatform.app`.
4. Confirm an **Apple Distribution** certificate is available to the team
   (Codemagic can create one from the API key if not).

### B. Codemagic dashboard (Phase 2)
1. Verify the App Store Connect API key integration name matches
   `integrations.app_store_connect: Aura Platform LLC` in `codemagic.yaml`
   **exactly** — if the UI name differs, update the YAML value to match.
2. Add a **secure** environment variable `FIREBASE_IOS_CONFIG_BASE64` =
   base64 of `ios/Runner/GoogleService-Info.plist` (gitignored; see
   `FIREBASE_SETUP.md`).
3. Start the `ios-testflight` workflow.

### C. Firebase Console (Phase 4 — required for iOS push to actually deliver)
1. Upload the APNs auth key (`.p8`) under
   *Project settings → Cloud Messaging → Apple app configuration*.
2. Ensure the iOS app entry uses bundle ID `org.auraplatform.app`.
3. Set `FIREBASE_PROJECT_ID`, `FIREBASE_CLIENT_EMAIL`, `FIREBASE_PRIVATE_KEY`
   in Railway (these power `FcmPushAdapter`).

### D. App Store Connect metadata
- Add an **App Review demo account** (review-safe credentials) so reviewers
  can sign in. Confirm that account works against the production backend.

### E. Real-device TestFlight validation (Phase 6 — cannot be simulated)
After the build is on TestFlight, on a physical iPhone:
1. Install the TestFlight build; log in with the review account.
2. Confirm an iOS device token registers (`GET /v1/devices/me` shows an
   active `IOS` device).
3. Receive a normal notification.
4. Receive a call/ring notification; confirm the incoming-call overlay.
5. Confirm a notification tap opens the correct route.
6. Log out → confirm the device token is revoked.
7. Reinstall + log in → confirm a fresh token re-registers.
